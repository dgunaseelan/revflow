Behind the Scenes
RevFlow Middleware Server

A dedicated server (similar to developer.salesforce.com's backend) that orchestrates the entire provisioning pipeline. Receives the signup form, calls the Salesforce Signup API, provisions integration users, and wires the new org to the customer's Slack workspace.

Provisioning Pipeline

Pre-warmed pool model (recommended): Middleware grabs a preconfigured templated org from the pool in seconds — already set up with Revenue Cloud via Salesforce Go — Revenue Cloud Initial Setup. Alternative: on-demand provisioning via Signup API (takes minutes).

Revenue Cloud Auto-Configuration

Uses a preconfigured templated org or Salesforce Go — Revenue Cloud Initial Setup to provision orgs with Revenue Cloud objects, pricing rules, approval workflows, and a default product catalog. No manual metadata deploy.

Leadership Org Edition & Cost
Enterprise edition org has a cost of $165-175/user/mo. Which edition should RevFlow provision for each customer?
Research Findings
Starter Suite ($25/user/mo) — No API access. Limited to 10 users. Cannot support RevFlow's API-driven backend. UNVIABLE.

Pro Suite ($100/user/mo) — 5,000 API calls/day. Has Flows, custom objects, Account access, AppExchange. May be sufficient for low-volume SMBs but needs load-testing validation against RevFlow's usage patterns.

Enterprise ($165-175/user/mo) — Unlimited API calls. Advanced workflow automation, territory management. Safest option but highest cost per org.

Open: Is there a lightweight SKU or internal-use edition that can be provisioned at reduced cost? Does SF Go support zero/reduced-cost provisioning for RevFlow?
Leadership Cost Absorption
Who subsidizes the Salesforce org infrastructure cost for free-tier customers? At scale (10K+ free users), this is the most expensive unresolved business question.
Options Under Consideration
(a) Salesforce subsidizes as customer acquisition cost
(b) Loss-led for a capped period (e.g. 90 days)
(c) Lightweight org edition to reduce per-org cost
(d) SF Go mechanism for zero/reduced-cost provisioning
Architecture Provisioning SLA
What is the Signup API provisioning latency? If org creation takes minutes, the pre-warmed pool model is mandatory for a self-serve freemium experience. Pool management adds operational complexity.
Architecture RevFlow Middleware Server — Scope & Hosting
The RevFlow Middleware Server orchestrates provisioning (org creation, user setup, Slack wiring) but is not in the runtime path — Slack calls SF APIs directly via MCP at runtime. Key questions: Where is it hosted? What is its availability SLA? How does it manage the pre-warmed org pool? What secrets/credentials does it store?
Architectural Considerations
Scope: Provisioning-only server. Handles signup → org creation → integration user setup → Slack workspace connection. Not involved in day-to-day RevFlow operations.

Analogous to: developer.salesforce.com's backend — receives signup forms, calls Signup API, provisions orgs, returns credentials.

Hosting options: Heroku, AWS, or Salesforce Hyperforce. Needs access to Signup API (org provisioning) and Slack API (workspace wiring).

Security: Stores Signup API credentials, org admin tokens (temporarily), and Slack app tokens. Requires vault/secrets management.

Technical View

Behind the Scenes
Integration User (API-Only)

The RevFlow Middleware creates a Salesforce Integration User with the Minimum Access – API Only Integrations profile. This user:

  • Cannot log in to Salesforce UI
  • Accesses data only via REST, SOAP, or Bulk API
  • Uses OAuth Client Credentials Flow for auth
  • Gets permissions via Permission Sets (not the profile)
JIT User Provisioning

The Middleware maps Slack workspace members to SF users via Just-In-Time (JIT) provisioning. Slack email → SF User. No SF login credentials are ever created for end users — at runtime, Slack calls SF APIs directly via Salesforce and Revenue Cloud MCP.

Open Questions for This Step

Leadership Sales Cloud License Requirement
RevFlow needs access to the Account object and platform items like Profiles, Flows, and custom objects. Is one Sales Cloud license required for platform access? Or can the Integration User license alone provide what's needed?
Architectural Consideration
The Integration User license provides API access but may not include standard objects like Account, Opportunity, or Product2 without additional licensing. A single Sales Cloud license might unlock the platform schema that RevFlow's API calls rely on. This needs validation with the Salesforce Licensing team.
Architecture Keeping Salesforce Invisible
How do we prevent any Salesforce UI from being accessible to end users?
Research: Integration User + Headless 360
Integration Users with the "Minimum Access – API Only Integrations" profile physically cannot log into the Salesforce UI. The profile is locked and non-editable.

This is now further validated by Salesforce Headless 360, announced April 15, 2026 at TrailblazerDX. Parker Harris asked: "Why should you ever log into Salesforce again?"

Headless 360 exposes the entire Salesforce platform as 60+ MCP tools and APIs — no browser required. This is the strongest architectural validation for RevFlow's invisible-Salesforce approach.

Three-layer invisibility:
1. API-Only Integration Users (no UI login)
2. Slack calls SF APIs via Salesforce & Revenue Cloud MCP (no SF URLs exposed)
3. Headless 360 platform direction (API is the UI)
Architecture Token Refresh for Provisioned Orgs
How does Seamless Auth handle token refresh for RevFlow-provisioned orgs? These orgs are not customer-initiated — does RevFlow act as the Connected App? Access tokens expire in ~2 hours; refresh tokens are indefinite.

Behind the Scenes
Seamless Auth Flow

Zero-config authentication between Slack and Salesforce. No custom OAuth flows needed. Seamless Auth handles:

  • Automatic user mapping (Slack email → SF user)
  • Server-side token storage
  • Access tokens (~2hr), refresh tokens (indefinite)
  • Workspace-to-org tenant mapping via Seamless Auth
Tenant Mapping (Middleware-Initiated)

The RevFlow Middleware registers the tenant mapping: workspace_id:T04ACME ↔ org_id:00D5e000001ABC. Once connected, all subsequent Slack interactions call Salesforce APIs directly through Salesforce and Revenue Cloud MCP tools — no middleware in the runtime path.

1. RevFlow Middleware Server — Scope & Hosting
Researched
The RevFlow Middleware Server orchestrates provisioning (org creation, user setup, Slack wiring) but is not in the runtime path — Slack calls SF APIs directly via MCP at runtime. Where is it hosted? What is its availability SLA? How does it manage the pre-warmed org pool?

2. Provisioning SLA
Needs Decision

3. How to Keep Salesforce Invisible
Researched
The core architectural challenge: how do we ensure end users never see, access, or know about the underlying Salesforce org?
Solution: Three-layer invisibility model

Layer 1 — Integration Users: The "Minimum Access – API Only Integrations" profile physically prevents UI login. The profile is locked/non-editable. OAuth Client Credentials flow for server-to-server only. When audited, Salesforce displays "Access Restricted for API Only Users."

Layer 2 — Slack via Salesforce & Revenue Cloud MCP: All API calls go through Salesforce and Revenue Cloud MCP tools directly from Slack. No Salesforce domain URLs ever appear in the Slack UI. Admin config and user management happen through Block Kit modals.

Layer 3 — Headless 360 Platform Direction: Announced April 15, 2026 at TrailblazerDX. Parker Harris: "Why should you ever log into Salesforce again?" The entire platform is now exposed as 60+ MCP tools and APIs. This is the strongest architectural validation for RevFlow's approach — Salesforce itself is moving toward headless.

Verdict: Architecturally sound. Integration Users + MCP-based API access + Headless 360 direction all align. No fundamental blocker.
